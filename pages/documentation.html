Usage

Permission Smells

P-Lint discovers Permission Smells as defined by the Android best practices guide. Many of these smells are defined below:

Application Targets

  • {uncompressedAPK}/app/AndroidManifest.xml: Looking for information specific to APK that is found here (Use cases: 2, 3, 4, 8-16)
  • {uncompressedAPK}/app/src/android/support/v4: Searching through java files for specific use cases. Using this directory as it seems to be where relevant app controlling java files are stored. (Use cases: 1, 5, 6, 7)

Use Cases

  • Use case 1: Searching for instances of the method call “shouldShowRequestPermissionRationale” in java files (Good)
  • Use case 2: Searching for dangerous permission CAMERA that could be replaced by an intent using Android’s camera application (Not bad or good)
  • Use case 3: Searching through manifest for dangerous permission SEND_SMS that could possibly be replaced by an intent using Android’s existing messaging service (Not bad or good)
  • Use case 4: Searching through manifest for dangerous permission CALL_PHONE that could possibly be replaced by an intent using Android’s existing phone application (Not bad or good)
  • Use case 5: APK does NOT initialize checkSelfPermission method even if permission requests are made (Bad)
  • Use case 6: APK does initialize checkSelfPermission method when permission requests are made (Good)
  • Use case 7: Checks to see if multiple permission requests are made in close proximity in the code, possibly overwhelming user with requests (Bad)
  • Use case 8: Custom permission is declared in manifest, could fail depending on its dependencies to dangerous permissions if they’re not granted (Not good or bad)
  • Use case 9: Permission declaration has a min SDK version set below 23, and no max SDK version. (Check that it works for BOTH permission models)
  • Use case 10: Permission declaration has min SDK version below 23 and max SDK version at or above 23 (Good)
  • Use case 11: Permission declaration has min SDK version at or above 23 (Should check how this permission is handled/used in older versions)
  • Use case 12: Permission declaration has max SDK version below 23 (should check to see how permission behavior occurs on new model)
  • Use case 13: Permission declaration has max SDK version at or above 23, but no min. SDK version (should check to see how it functions in old model)
  • Use case 14: APK’s min SDK version is below 23, and max SDK version is at or above 23 (Should check how it functions on BOTH models)
  • Use case 15: APK’s max SDK version is below 23 (Does not follow M-Permission model, disregard the results for this APK)
  • Use case 16: APK’s min SDK version is at or above 23 (Only needs to check that it adheres to M-Permission model)
